# API Key Management Portal — Implementation Plan

## Context

The Finance API (`FsMathFunctions.Api`) currently validates requests against a **single** `API_KEY`
environment variable. This plan expands the system to support multiple users, each managing their
own API keys via a web portal.

---

## Goals

- Users register with their **email address** as the unique identifier
- Authenticated users can generate, view, and revoke their own API keys
- An admin role can see all users and their keys
- The Finance API validates incoming keys against a **shared PostgreSQL database**
- The portal UI uses **Microsoft Fluent UI v9** (React)
- Auth is **custom JWT** (email + bcrypt password) — no external identity provider
- Deployable as **Docker containers** on **Azure Container Apps (ACA)**

## Database Schema

```
users
  id            UUID        PK
  email         TEXT        UNIQUE NOT NULL
  password_hash TEXT        NOT NULL
  role          TEXT        NOT NULL  DEFAULT 'user'   -- 'user' | 'admin'
  created_at    TIMESTAMPTZ NOT NULL

api_keys
  id          UUID        PK
  user_id     UUID        NOT NULL  FK → users.id
  label       TEXT        NOT NULL
  key_hash    TEXT        NOT NULL  -- SHA-256 of the raw key (raw key never stored)
  key_prefix  TEXT        NOT NULL  -- first 8 chars, shown in UI
  created_at  TIMESTAMPTZ NOT NULL
  revoked_at  TIMESTAMPTZ NULL
```

## Portal Backend Endpoints

```
POST  /auth/register        { email, password }   → 201
POST  /auth/login           { email, password }   → { token, expiresAt }

GET   /api/keys             JWT required          → ApiKeyDto[]
POST  /api/keys             { label }             → { id, rawKey, prefix, label }
DELETE /api/keys/{id}       JWT required          → 204

GET   /admin/users          JWT, admin role       → UserSummaryDto[]
DELETE /admin/users/{id}    JWT, admin role       → 204
```

## Key Generation (Security Design)

```
generate():
  rawKey    = base64url( RandomNumberGenerator.GetBytes(32) )  -- 43-char URL-safe string
  keyPrefix = rawKey[0..7]                                     -- shown in UI, safe to store
  keyHash   = SHA-256(rawKey)                                  -- stored in DB, never raw

  persist { userId, label, keyHash, keyPrefix, createdAt }
  return rawKey to caller ONCE — not retrievable again
```

**Finance API validation** (updated middleware):
```
incoming = X-API-Key request header
hash     = SHA-256(incoming)
valid    = db.ApiKeys.Any(k => k.KeyHash == hash && k.RevokedAt == null)
```

## Azure Deployment (ACA — recommended starting point)

- One **Container Apps Environment**  
- Three container apps: `finance-api`, `portal`, `portal-ui`
- **Azure Database for PostgreSQL Flexible Server** (managed, no patching)
- **Azure Container Registry** for Docker images
- **Managed Identity** for DB credentials (no passwords in app config)
- Ingress: `portal-ui` public, `portal` public on `/auth` and `/api`, `finance-api` internal only

Upgrade path to **AKS**: possible without changing application code — the Dockerfiles and
connection strings remain identical.

3. `FsMathFunctions.Portal` — auth + key management backend
4. `docker-compose.yml` — local multi-container dev environment
5. `FsMathFunctions.Portal.UI` — React + Fluent UI v9 frontend
6. `infra/` Bicep — ACA deployment infrastructure
